# 처방 변화 레이더 설계 노트

## 목적

처방 변화 레이더는 새로 분석한 처방과 같은 환자의 최근 관련 처방을 비교해 약품 구성과 복약 일정의 객관적 차이를 보여준다. 이 기능은 의학적 판단이나 복용 시작·중단 지시를 생성하지 않는다.

## BCE 책임 분리

- Boundary: `PrescriptionChangeRadarUI`가 비교 결과와 안전 안내를 표시한다.
- Control: 프론트 `CheckPrescriptionChange`가 비교 요청을 전송하고, 백엔드 `CheckPrescriptionChange`가 이전 처방 조회와 비교를 수행한다.
- Entity: 프론트 `PrescriptionChangeRadar`와 백엔드 `schemas/prescription_change.py`가 비교 결과 계약을 정의한다.
- 저장 Entity: 기존 `_SavedMedication`은 변경하지 않고 읽기 전용 비교 기준으로 사용한다.

## 비교 규칙

1. 환자 해시가 같은 저장 정보만 조회한다.
2. 현재 조제일자 기준 최근 90일 안의 처방만 비교 후보로 사용한다. 이 기간은 비교 유효기간이며 저장 데이터 삭제 주기와는 별개다.
3. 모든 후보에 대해 동일 품목, 동일 성분, 치료 맥락 순으로 관련성 점수를 계산한다.
4. 관련성이 확인된 후보 중 점수가 가장 높은 처방을 선택하고, 점수가 같으면 더 최근 처방을 선택한다.
5. 관련 후보가 없으면 `unrelated`, 기록이 90일을 넘었으면 `expired`로 응답하고 약품 추가·미확인 변화를 생성하지 않는다.
6. 같은 약품은 공공데이터 품목 식별자(`item_seq`)를 우선 사용하고, 식별자가 없으면 정규화한 약품명으로 연결한다.
7. 현재 처방에만 있는 약은 `added`, 이전 처방에만 있는 약은 `missing`으로 분류한다.
8. 같은 약의 1회 투약량, 1일 횟수, 총 투약일이 달라지면 `schedule_changed`로 분류한다.
9. `missing`은 복용 중단을 뜻하지 않으며 UI에서 “이번 처방에서 확인되지 않음”으로 표시한다.

## 실패 격리

처방 변화 비교는 처방 분석의 보조 기능이다. 비교 API가 실패해도 약품 상세 분석과 복약 일정 저장은 계속 진행한다.

## 현재 한계

- 같은 날짜에 서로 다른 처방이 여러 번 발행된 경우 별도 처방 ID가 없어 하나의 날짜 묶음으로 취급한다.
- 치료 맥락은 저장된 효능 문구의 제한된 핵심어를 사용하므로 모든 질환군을 완전히 판정하지는 못한다.
- 주성분 정보가 로컬 허가정보에 없으면 약품명 괄호 안 성분명을 보조 근거로 사용한다.
- 향후 처방 ID 또는 병원·처방전 식별자가 추가되면 날짜 묶음보다 정확한 비교가 가능하다.
