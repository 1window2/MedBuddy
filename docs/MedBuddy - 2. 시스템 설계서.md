# (D1) 클래스 설계서

## **1\. 개요**

MedBuddy는 사용자가 모바일 환경에서 의약품 정보 또는 처방전 관련 텍스트를 입력하면, 이를 분석하여 약 정보를 조회하고, 이해하기 쉬운 안내를 제공하며, 필요 시 개인 약통에 저장할 수 있도록 지원하는 시스템이다. 현재 백엔드는 FastAPI 기반으로 구성되어 있으며, OCR 결과 텍스트 정제, 공공 의약품 API 조회, AI 기반 복약 안내 생성, 저장 기능을 중심으로 구현되어 있다.

본 클래스 설계서는 MedBuddy 시스템을 구성하는 주요 클래스의 책임, 속성, 메서드 및 클래스 간 관계를 정의한다. 또한 처방전 구조화 기능의 정확도와 안정성을 높이기 위해, 규칙 기반 파싱을 기본으로 하고 필요 시 LLM을 복구용 보조 모듈로 활용하는 확장 구조까지 함께 포함한다.

## **3\. 클래스 상세 설계**

### **3.1 `SavedMedication`**

**설명**  
 사용자가 약통에 저장한 약 정보를 데이터베이스에 보관하기 위한 SQLAlchemy 모델 클래스이다. 테이블명은 `saved_medications`이며, 약 이름, 효능, 사용법, 주의사항, AI 안내 문구를 저장한다.

**속성**

| 속성명 | 타입 | 설명 |
| ----- | ----- | ----- |
| `id` | Integer | 기본키 |
| `item_name` | String | 약 이름 |
| `efficacy` | String | 약 효능 |
| `use_method` | String | 복용 방법 |
| `warning_message` | String | 주의사항 |
| `ai_guide` | String / Nullable | AI가 생성한 사용자 친화적 복약 안내 |

**주요 역할**

* 저장 약 정보 영속화  
* 저장/조회/삭제 API의 데이터 원본 제공

---

### **3.2 `MedicationRequest`**

**설명**  
 클라이언트가 약 식별 요청 시 전달하는 입력 모델이다. 현재는 `extracted_text` 하나를 입력받는다.

**속성**

| 속성명 | 타입 | 설명 |
| ----- | ----- | ----- |
| `extracted_text` | Optional\[str\] | 앱에서 ML Kit 등으로 추출한 OCR 텍스트 |

**주요 역할**

* 약 조회 API 입력 형식 검증

---

### **3.3 `DrugInfo`**

**설명**  
 조회된 개별 약 정보를 표현하는 응답용 데이터 모델이다. 공공 API에서 받은 원본 정보를 가공하여 응답에 포함한다.

**속성**

| 속성명 | 타입 | 설명 |
| ----- | ----- | ----- |
| `item_name` | str | 약 이름 |
| `efficacy` | str | 효능 |
| `use_method` | str | 사용법 |
| `warning_message` | str | 주의사항 |
| `ai_guide` | Optional\[str\] | AI가 생성한 쉬운 설명 |

**주요 역할**

* 공공 API 결과를 표준화된 형태로 표현  
* 최종 사용자 응답 데이터 단위 제공

---

### **3.4 `SavedMedicationCreate`**

**설명**  
 사용자가 특정 약 정보를 약통에 저장할 때 사용하는 입력 모델이다. 저장 시 필요한 필드를 검증한다.

**속성**

| 속성명 | 타입 | 설명 |
| ----- | ----- | ----- |
| `item_name` | str | 약 이름 |
| `efficacy` | str | 효능 |
| `use_method` | str | 사용법 |
| `warning_message` | str | 주의사항 |
| `ai_guide` | Optional\[str\] | AI 안내 문구 |

---

### **3.5 `MedicationResponse`**

**설명**  
 약 식별 및 조회 결과를 클라이언트에 반환하는 최종 응답 모델이다. 성공 여부, 메시지, 약 정보 리스트를 포함한다.

**속성**

| 속성명 | 타입 | 설명 |
| ----- | ----- | ----- |
| `success` | bool | 처리 성공 여부 |
| `message` | str | 결과 메시지 |
| `data` | List\[`DrugInfo`\] | 조회된 약 정보 목록 |

---

### **3.6 `OCRService`**

**설명**  
 OCR 엔진 자체를 수행하는 클래스라기보다, 현재 구조에서는 클라이언트에서 전달한 OCR 결과 문자열을 후처리하는 서비스 클래스이다. 줄바꿈 제거 및 문자열 정제 후 검색 키워드를 생성한다.

**메서드**

| 메서드명 | 입력 | 반환 | 설명 |
| ----- | ----- | ----- | ----- |
| `process_text(raw_text)` | `str` | `str` | OCR 결과 문자열을 정제하여 검색용 키워드 반환 |

**주요 역할**

* OCR 노이즈 최소화  
* 약 조회 전처리 담당

---

### **3.7 `DrugService`**

**설명**  
 공공데이터포털 의약품 API와 통신하는 서비스 클래스이다. 설정값에서 API 키와 기본 URL을 읽고, 약 이름을 기반으로 외부 API를 호출한 후 `DrugInfo` 목록으로 변환한다.

**속성**

| 속성명 | 타입 | 설명 |
| ----- | ----- | ----- |
| `api_key` | str | 공공 API 인증키 |
| `basic_url` | str | 공공 API(의약품개요정보) 기본 주소 |
| `advanced_url` | str | 공공 API(의약품제품허가정보) 기본 주소 |

**메서드**

| 메서드명 | 입력 | 반환 | 설명 |
| ----- | ----- | ----- | ----- |
| `fetch_drug_info(drug_name)` | `str` | `list[DrugInfo]` | 약 이름으로 공공 API 조회 후 결과 반환 |

**주요 역할**

* 외부 의약품 데이터 수집  
* 응답 파싱 및 도메인 모델 변환

---

### **3.8 `Medication Router`**

**설명**  
 FastAPI `APIRouter`를 이용해 구현된 제어 계층이다. `OCRService`, `DrugService`, DB 세션에 의존하며 약 조회, 저장, 목록 조회, 삭제 기능을 제공한다. `/identify`, `/save`, `/list`, `/delete/{drug_id}` 엔드포인트가 구현되어 있다.

**주요 기능**

| 엔드포인트 | 설명 |
| ----- | ----- |
| `POST /identify` | OCR 텍스트 기반 약 정보 조회 및 AI 안내 생성 |
| `POST /save` | 약통에 약 정보 저장 |
| `GET /list` | 저장된 약 목록 조회 |
| `DELETE /delete/{drug_id}` | 저장된 약 삭제 |

**주요 역할**

* 요청 수신 및 입력 검증  
* 서비스 호출 조정  
* DB 처리 및 응답 반환

---

### **3.9 `PrescriptionParser` 기능군**

**설명**  
처방전 OCR 결과를 줄 단위로 입력받아 환자명, 처방일자, 약 목록을 구조화된 데이터 형태로 반환하는 1차 규칙 기반 파싱 모듈이다. 현재는 normalize\_text, normalize\_date, extract\_patient\_name, extract\_prescription\_date, parse\_medication\_line, parse\_prescription 등의 함수로 구현되어 있으며, 약 한 줄에 대해 약 이름, 1회 투약량, 1일 복용 횟수, 총 복용 일수를 추출하는 정규식 기반 로직을 포함한다. 또한 본 설계에서는 파싱 결과가 불완전할 경우 ParsingValidator를 통해 실패 여부를 판정하고, 필요 시 LLMRecoveryService를 통해 복구를 수행하는 구조를 고려한다.

**주요 함수**

| 함수명 | 설명 |
| ----- | ----- |
| `normalize_text(text)` | OCR 문자열 정규화 |
| `normalize_date(text)` | 날짜 형식 표준화 |
| `extract_patient_name(line)` | 환자명 추출 |
| `extract_prescription_date(line)` | 처방 날짜 추출 |
| `parse_medication_line(line)` | 약 정보 1행 파싱 |
| `parse_prescription(lines)` | 전체 처방전 구조화 |

**주요 역할**

* 처방전 텍스트 구조화  
* 향후 복약 일정 생성 및 약 상호작용 분석의 입력 데이터 제공

---

### **3.10 `ParsingValidator`**

**설명**  
 규칙 기반 파서의 결과가 충분히 완전하고 신뢰할 수 있는지를 검사하는 검증 클래스이다. 약 이름 미검출, 복용 횟수 또는 복용 기간 누락, 날짜 추출 실패, 필수 필드 누락 등의 조건을 바탕으로 파싱 성공 여부를 판단한다.

**메서드**

| 메서드명 | 입력 | 반환 | 설명 |
| ----- | ----- | ----- | ----- |
| `validate(parsed_result)` | dict / Prescription | bool | 파싱 결과가 유효한지 판정 |
| `check_missing_fields(parsed_result)` | dict / Prescription | list\[str\] | 누락된 필수 필드 목록 반환 |
| `calculate_confidence(parsed_result)` | dict / Prescription | float | 파싱 결과의 신뢰도 점수 계산 |

**주요 역할**

* 파싱 결과의 품질 검증  
* LLM 복구 호출 여부 결정  
* 잘못된 구조화 결과의 후속 오류 방지

---

### **3.11 `LLMRecoveryService`**

**설명**  
 규칙 기반 파싱이 실패하거나 결과가 불완전한 경우, OCR 텍스트를 바탕으로 구조화 결과를 복구하는 보조 서비스 클래스이다. 본 클래스는 기본 파서 대신 항상 사용되는 것이 아니라, ParsingValidator가 실패를 판정한 경우에만 예외적으로 호출된다. 또한 개인정보 보호를 위해 환자명 등 민감 정보는 마스킹 또는 제거한 후 복구 요청을 수행한다.

**메서드**

| 메서드명 | 입력 | 반환 | 설명 |
| ----- | ----- | ----- | ----- |
| `recover_structure(raw_text)` | str | dict / Prescription | OCR 원문을 바탕으로 구조화 결과 복구 |
| `sanitize_text(raw_text)` | str | str | 민감 정보를 제거하거나 마스킹한 텍스트 반환 |
| `normalize_llm_output(result)` | dict | dict / Prescription | LLM 출력 결과를 시스템 형식에 맞게 정규화 |

**주요 역할**

* 파싱 실패 시 구조 복구  
* 민감 정보 최소화 후 LLM 호출  
* JSON 형식의 일관된 출력 보장

---

### **3.12 `FastAPI App`**

**설명**  
 애플리케이션 시작점으로, 환경변수 로드, Gemini 초기화, DB 테이블 생성, FastAPI 앱 생성, 라우터 등록을 담당한다. `main.py`에서 구현되어 있다. 또한 전체 처리 흐름에서 규칙 기반 파싱, 파싱 결과 검증, 필요 시 LLM 복구 모듈이 순차적으로 연동될 수 있도록 시스템 구성을 담당한다.

**주요 역할**

* 시스템 초기화  
* 라우팅 구성  
* 서버 실행 진입점 제공

## **4\. 클래스 간 관계**

현재 MedBuddy 클래스 관계는 다음과 같이 정리할 수 있다.

Client  
 ↓  
Medication Router  
 ├─ uses → OCRService  
 ├─ uses → PrescriptionParser  
 ├─ uses → ParsingValidator  
 ├─ if validation fails → LLMRecoveryService  
 ├─ uses → DrugService  
 ├─ uses → SavedMedicationCreate / MedicationRequest  
 ├─ returns → MedicationResponse / DrugInfo / Prescription  
 └─ accesses → SavedMedication (DB)

FastAPI App  
 └─ registers → Medication Router

사용자의 요청은 먼저 Medication Router에서 수신되며, `OCRService`가 텍스트를 정제한 후 `PrescriptionParser`가 이를 1차 규칙 기반 방식으로 구조화한다. 이후 `ParsingValidator`가 파싱 결과의 완전성과 신뢰도를 검사하며, 결과가 불완전하다고 판단되면 `LLMRecoveryService`가 예외적으로 호출되어 구조 복구를 수행한다. 약 정보 조회 기능에서는 `DrugService`가 외부 의약품 API를 호출하고, 결과는 `DrugInfo` 및 `MedicationResponse` 형태로 반환된다. 저장 기능에서는 `SavedMedicationCreate` 입력을 받아 `SavedMedication` 엔티티로 변환한 뒤 데이터베이스에 저장한다.

## **5\. 설계 특징**

첫째, 입력 검증용 스키마, 비즈니스 서비스, DB 모델, API 제어부가 분리되어 있어 역할 분담이 비교적 명확하다. 둘째, OCR 후처리와 의약품 조회를 별도 서비스로 분리하여 향후 OCR 엔진 교체나 API 변경에 대응하기 쉽다. 셋째, 처방전 파싱 기능이 별도 모듈로 존재하므로, 향후 “사진 기반 처방전 구조화 → 복약 일정 추천 → 안전성 검사” 흐름으로 확장할 수 있다. 다만 현재 파서는 함수 중심이므로, 설계 문서에서는 향후 `PrescriptionParser` 클래스로 리팩토링 가능한 구조라고 명시하면 더 자연스럽다. 넷째, 규칙 기반 파싱 이후 별도의 검증 계층과 LLM 기반 복구 계층을 두어, 다양한 처방전 형식이나 OCR 오류로 인해 파싱이 실패하는 상황에도 보다 안정적으로 대응할 수 있도록 설계하였다. 다섯째, LLM은 기본 경로가 아니라 예외적 복구 경로에서만 사용되므로 성능 저하와 비용 증가를 최소화할 수 있다.

## **6\. 향후 확장 방향**

현재 구현을 기준으로 D1 설계서에 아래 내용을 “확장 예정 클래스”로 넣으면 좋다.

| 클래스명 | 설명 |
| ----- | ----- |
| `Prescription` | 처방전 전체 정보를 객체화 |
| `Medicine` | 개별 약 정보를 객체화 |
| `ParsingValidator` | 파싱 결과의 완전성과 신뢰도를 검사 |
| `LLMRecoveryService` | 파싱 실패 시 구조 복구 수행 |
| `MedicationScheduler` | 복약 시간표 생성 |
| `InteractionChecker` | 약물 상호작용 검사 |
| `RecommendationService` | 사용자 맞춤 복약/생활 가이드 추천 |

이 확장 방향은 현재 저장/조회 중심 구조를 넘어, MedBuddy를 실제 복약 보조 시스템으로 발전시키기 위한 설계적 근거가 된다. 현재 저장 모델과 처방전 파서가 이미 존재하므로, 위 확장은 완전히 새로운 구조라기보다 기존 구현의 자연스러운 심화로 볼 수 있다.

---

## **7\. 결론**

MedBuddy의 현재 클래스 구조는 약 정보 조회 및 저장 기능을 중심으로 기본적인 계층 분리가 이루어져 있으며, OCR 텍스트 처리, 외부 API 조회, DB 저장, 응답 모델링이 각각 독립된 책임을 가진다. 또한 별도의 처방전 파싱 모듈이 존재하여 향후 구조화된 복약 관리 시스템으로 확장할 수 있는 기반을 갖추고 있다. 따라서 본 설계는 현재 구현 상태를 반영하면서도 이후 고도화를 수용할 수 있는 클래스 구조라고 판단된다.

# (D3) 컴포넌트 설계서

## **1\. 개요**

본 문서는 MedBuddy 시스템의 주요 컴포넌트를 정의하고, 각 컴포넌트의 역할과 상호작용 구조를 설계하는 것을 목적으로 한다.

MedBuddy는 사용자가 입력한 처방전 또는 약봉투 이미지를 기반으로 OCR을 수행하고, 추출된 텍스트를 분석하여 약 정보와 복용 정보를 구조화한 뒤, 사용자에게 직관적인 형태로 제공하는 시스템이다. 이를 위해 본 시스템은 이미지 입력, 텍스트 추출, 처방 정보 파싱, 결과 검증, 복구 처리, 약 정보 조회, 저장 및 추천 기능을 각각 독립된 컴포넌트로 분리하여 설계한다.

본 설계는 기능 간 결합도를 낮추고 유지보수성과 확장성을 높이기 위해, 각 컴포넌트가 명확한 책임을 가지도록 구성하였다.

---

## **2\. 컴포넌트 목록**

| 컴포넌트명 | 설명 |
| ----- | ----- |
| Image Input Component | 사용자의 이미지 촬영 및 업로드를 처리 |
| OCR Component | 입력 이미지에서 텍스트를 추출 |
| Text Preprocessing Component | OCR 결과 텍스트를 정제 |
| Prescription Parsing Component | 약 이름, 복용 횟수, 기간 등 처방 정보를 구조화 |
| Parsing Validation Component | 파싱 결과의 완전성과 정확성을 검증 |
| LLM Recovery Component | 파싱 실패 시 구조화 결과를 복구 |
| Drug Information Component | 외부 API를 통해 의약품 상세 정보 조회 |
| Medication Storage Component | 저장된 약 정보를 관리 |
| Recommendation Component | 복약 기반 건강 추천 정보 생성 |
| Output Component | 결과를 사용자 화면 및 음성 형태로 제공 |

---

## **3\. 컴포넌트 상세 설계**

### **3.1 Image Input Component**

**설명**  
 사용자가 모바일 환경에서 처방전 또는 약봉투 이미지를 입력하는 컴포넌트이다.

**입력**

* 카메라 촬영 이미지  
* 갤러리 업로드 이미지

**출력**

* 이미지 파일 데이터

**주요 역할**

* 이미지 입력 처리  
* 촬영 및 업로드 기능 제공

---

### **3.2 OCR Component**

**설명**  
 입력된 이미지로부터 텍스트를 추출하는 컴포넌트이다. 본 시스템에서는 Google ML Kit 기반 OCR을 사용한다.

**입력**

* 이미지 파일 데이터

**출력**

* OCR 원문 텍스트

**주요 역할**

* 이미지 내 문자 인식  
* 텍스트 데이터 생성

---

### **3.3 Text Preprocessing Component**

**설명**  
 OCR 결과 텍스트의 줄바꿈, 공백, 불필요한 문자 등을 정리하여 후속 파싱에 적합한 형태로 정제하는 컴포넌트이다.

**입력**

* OCR 원문 텍스트

**출력**

* 정제된 텍스트

**주요 역할**

* 텍스트 정규화  
* OCR 노이즈 감소  
* 파싱 입력 품질 향상

---

### **3.4 Prescription Parsing Component**

**설명**  
 정제된 텍스트를 바탕으로 약 이름, 복용 시간, 복용 횟수, 복용 기간 등의 정보를 추출하여 구조화하는 핵심 컴포넌트이다. 본 컴포넌트는 1차적으로 규칙 기반 파싱을 수행한다.

**입력**

* 정제된 텍스트

**출력**

* 구조화된 처방 정보(JSON)

**주요 역할**

* 약 정보 추출  
* 복용 정보 추출  
* 구조화 데이터 생성  
* 레이아웃 기반 정보 해석

이 컴포넌트는 요구사항의 FR-5, FR-6, FR-7, FR-8과 직접적으로 연결된다.

---

### **3.5 Parsing Validation Component**

**설명**  
 Prescription Parsing Component의 결과가 충분히 완전하고 신뢰할 수 있는지 검사하는 컴포넌트이다.

**입력**

* 구조화된 처방 정보

**출력**

* 검증 결과(Boolean 또는 신뢰도 점수)  
* 누락 필드 목록

**주요 역할**

* 필수 정보 존재 여부 확인  
* 파싱 성공 여부 판정  
* 복구 처리 필요 여부 결정

---

### **3.6 LLM Recovery Component**

**설명**  
 규칙 기반 파싱이 실패하거나 결과가 불완전할 경우, OCR 텍스트를 기반으로 구조화 결과를 복구하는 보조 컴포넌트이다. 본 컴포넌트는 기본 경로가 아닌 예외 경로에서만 사용된다.

**입력**

* OCR 원문 또는 정제 텍스트  
* 검증 실패 정보

**출력**

* 복구된 구조화 데이터

**주요 역할**

* 파싱 실패 시 정보 복구  
* 구조화 결과 보정  
* 다양한 형식의 처방전에 대한 대응력 향상

또한 개인정보 보호 요구사항을 반영하여, 민감한 정보는 복구 처리 전에 최소화 또는 마스킹하는 방식으로 설계한다. 이는 요구사항의 FR-12, FR-13과도 연결된다.

---

### **3.7 Drug Information Component**

**설명**  
 추출된 약 이름을 기준으로 외부 의약품 API를 호출하여 상세 정보를 조회하는 컴포넌트이다.

**입력**

* 약 이름

**출력**

* 약 효능, 복용 방법, 주의사항

**주요 역할**

* 의약품 상세 정보 조회  
* 사용자 이해를 돕는 정보 제공

---

### **3.8 Medication Storage Component**

**설명**  
 사용자가 확인한 약 정보를 저장하고, 이후 다시 조회하거나 삭제할 수 있도록 관리하는 컴포넌트이다.

**입력**

* 저장할 약 정보

**출력**

* 저장 결과  
* 저장 목록

**주요 역할**

* 약 정보 저장  
* 저장 목록 조회  
* 삭제 기능 제공

---

### **3.9 Recommendation Component**

**설명**  
 사용자의 복약 정보를 바탕으로 운동, 식습관 등 건강 관리 관련 추천 정보를 생성하는 컴포넌트이다.

**입력**

* 구조화된 약 정보  
* 사용자 상태 정보

**출력**

* 건강 관리 추천 정보

**주요 역할**

* 약물 기반 건강 추천  
* 약물 조합 고려  
* 사용자 맞춤 추천

이 컴포넌트는 FR-14, FR-15, FR-16과 연결된다.

---

### **3.10 Output Component**

**설명**  
 시스템의 분석 결과를 사용자에게 시각적 또는 음성 형태로 제공하는 컴포넌트이다.

**입력**

* 구조화된 약 정보  
* 추천 정보

**출력**

* 화면 표시 결과  
* 음성 안내 결과

**주요 역할**

* 정보 시각화  
* 음성 안내 제공  
* 사용자 친화적 결과 전달

이 컴포넌트는 FR-9, FR-10, FR-11과 연결된다.

---

## **4\. 컴포넌트 간 관계**

MedBuddy의 주요 컴포넌트 관계는 다음과 같다.

Image Input Component  
  ↓  
OCR Component  
  ↓  
Text Preprocessing Component  
  ↓  
Prescription Parsing Component  
  ↓  
Parsing Validation Component  
  ├─ 성공 → Drug Information Component → Output Component  
  └─ 실패 → LLM Recovery Component → Drug Information Component → Output Component

구조화된 결과  
  ├─ Medication Storage Component  
  └─ Recommendation Component → Output Component

설명하면, 사용자가 입력한 이미지는 OCR을 통해 텍스트로 변환되고, 텍스트 정제 후 처방 정보 파싱이 수행된다. 이후 검증 컴포넌트가 결과를 확인하여 정상적인 경우 바로 약 정보 조회와 결과 출력으로 연결되며, 실패한 경우에는 LLM 복구 컴포넌트를 거쳐 결과를 보완한다. 최종 구조화된 정보는 저장 기능과 추천 기능에도 활용된다.

## **1\. 개요**

본 문서는 MedBuddy 시스템을 구성하는 각 컴포넌트 간의 인터페이스를 정의하고, 데이터 흐름 및 상호작용 방식을 명확히 하는 것을 목적으로 한다.

MedBuddy 시스템은 이미지 입력부터 OCR, 텍스트 처리, 처방 정보 파싱, 검증, 복구, 약 정보 조회, 저장 및 추천 기능까지 여러 컴포넌트로 구성되어 있으며, 각 컴포넌트 간의 인터페이스를 명확히 정의함으로써 시스템의 결합도를 낮추고 확장성과 유지보수성을 향상시키고자 한다.

본 시스템은 다음과 같은 주요 인터페이스 흐름을 가진다.

Image Input → OCR → Text Preprocessing → Parsing → Validation → (LLM Recovery) → Drug Info → Output

각 컴포넌트는 입력과 출력을 명확히 정의하여 독립적으로 동작할 수 있도록 설계하였다.

## **3\. 인터페이스 상세 정의**

---

### **3.1 Image Input → OCR Interface**

**설명**  
 사용자가 입력한 이미지 데이터를 OCR 컴포넌트로 전달한다.

**입력 데이터**

* image: 이미지 파일 (jpg, png 등)

**출력 데이터**

* text: OCR 결과 문자열

---

### **3.2 OCR → Text Preprocessing Interface**

**설명**  
 OCR 결과를 텍스트 정제 컴포넌트로 전달한다.

**입력 데이터**

* raw\_text: OCR 원문 텍스트

**출력 데이터**

* cleaned\_text: 정제된 텍스트

---

### **3.3 Text Preprocessing → Parsing Interface**

**설명**  
 정제된 텍스트를 파싱 컴포넌트로 전달하여 구조화한다.

**입력 데이터**

* cleaned\_text: 정제된 텍스트

**출력 데이터 (예시 JSON)**

{  
 "patient\_name": "김홍익",  
 "date": "2026-03-31",  
 "medicines": \[  
   {  
     "name": "아목시실린",  
     "dose": "1회 1정",  
     "frequency": "1일 3회",  
     "duration": "5일"  
   }  
 \]  
}  
---

**설명**  
 파싱 결과의 완전성을 검증한다.

**입력 데이터**

* parsed\_result: 구조화된 데이터

**출력 데이터**

* is\_valid: Boolean  
* missing\_fields: 리스트  
* confidence\_score: float

---

### **3.5 Validation → LLM Recovery Interface (조건부)**

**설명**  
 파싱 결과가 불완전할 경우 LLM을 통해 복구를 수행한다.

**입력 데이터**

* sanitized\_text: 민감정보 제거된 텍스트  
* error\_info: 누락 정보

**출력 데이터**

* recovered\_result: 보정된 구조화 데이터

---

### **3.6 Parsing/Recovery → Drug Information Interface**

**설명**  
 추출된 약 이름을 기반으로 외부 API를 호출한다.

**입력 데이터**

* medicine\_names: 약 이름 리스트

**출력 데이터**

* drug\_info\_list

\[  
 {  
   "name": "아목시실린",  
   "efficacy": "...",  
   "usage": "...",  
   "warning": "..."  
 }  
\]  
---

### **3.7 Drug Information → Output Interface**

**설명**  
 최종 결과를 사용자에게 전달한다.

**입력 데이터**

**출력 데이터**

* 음성 안내 데이터

### **3.8 Output → Storage Interface**

**설명**  
 사용자가 선택한 약 정보를 저장한다.

**입력 데이터**

**입력 데이터**

* prescription\_data

**출력 데이터**

* recommendation\_data

## **5\. 주요 처리 흐름**

MedBuddy의 주요 처리 흐름은 다음과 같다.

1. 사용자가 모바일 앱에서 처방전 이미지를 촬영하거나 업로드한다.  
2. 클라이언트는 이미지를 OCR 처리 또는 서버 전송 준비 상태로 만든다.  
3. 서버는 텍스트를 정제한 뒤 Prescription Parser를 통해 1차 구조화를 수행한다.  
4. Parsing Validator가 결과를 검증한다.  
5. 검증 실패 시 LLM Recovery Service가 예외적으로 호출되어 결과를 복구한다.  
6. 추출된 약 이름을 기반으로 Drug Information Service가 외부 의약품 API를 호출한다.  
7. 최종 결과는 사용자에게 화면 및 음성 형태로 제공된다.  
8. 사용자가 원할 경우 결과를 저장하거나 추천 기능으로 확장한다.

이를 흐름도로 표현하면 다음과 같다.

사용자 입력  
  ↓  
모바일 클라이언트  
  ↓  
서버 요청  
  ↓  
텍스트 전처리  
  ↓  
규칙 기반 파싱  
  ↓  
검증  
  ├─ 성공 → 약 정보 조회 → 결과 출력  
  └─ 실패 → LLM 복구 → 약 정보 조회 → 결과 출력  
                        ↓  
                 저장 / 추천 기능

## **6\. 아키텍처 설계 특징**

### **6.1 계층 분리 구조**

클라이언트, 서버, 저장소, 외부 연동을 분리하여 각 영역의 책임을 명확히 하였다. 이를 통해 특정 계층의 변경이 전체 시스템에 미치는 영향을 줄일 수 있다.

### **6.2 규칙 기반 \+ 예외 복구 구조**

기본적으로는 규칙 기반 파싱을 중심으로 동작하게 설계하고, 파싱 실패 시에만 LLM 복구를 적용하는 구조를 채택하였다. 이 방식은 성능과 안정성을 동시에 고려한 구조이다.

### **6.3 보안 및 개인정보 최소화 반영**

처방전 이미지에는 민감 정보가 포함될 수 있으므로, 가능한 경우 로컬 처리와 정보 최소화 원칙을 적용하고, 서버 및 외부 연동 단계에서는 필요한 데이터만 사용하도록 설계하였다.

### **6.4 확장 가능한 모듈 구조**

Recommendation Service, 음성 안내, 상호작용 검사 등 향후 기능을 독립 모듈로 확장할 수 있도록 아키텍처를 구성하였다.

### **6.5 유지보수 용이성**

OCR, 파싱, 검증, 복구, 저장, 추천 기능을 각각 분리하여 특정 기능 개선이나 교체가 다른 기능에 미치는 영향을 줄일 수 있도록 하였다.

## **7\. 요구사항과의 연계성**

본 아키텍처 설계는 요구사항 분석서의 주요 기능 요구사항과 비기능 요구사항을 다음과 같이 반영한다.

* FR-1, FR-2: 클라이언트 계층의 이미지 입력 기능  
* FR-3, FR-4: OCR 처리 및 텍스트 추출 구조  
* FR-5 \~ FR-8: 서버 계층의 파싱 및 검증 구조  
* FR-9 \~ FR-11: 출력 계층과 음성 안내 기능  
* FR-12, FR-13: 로컬 처리 및 데이터 최소화 중심 보안 구조  
* FR-14 \~ FR-16: Recommendation Service 기반 확장 구조  
* NFR-2, NFR-3: 노인 친화적 UI 및 접근성 고려  
* NFR-4, NFR-5: 개인정보 보호 및 저장 최소화 반영  
* NFR-6: 파싱 검증 및 복구 구조를 통한 신뢰성 확보

## **8\. 결론**

본 아키텍처 설계는 MedBuddy 시스템의 핵심 기능을 계층적으로 분리하고, 각 기능을 독립적인 모듈로 구성함으로써 확장성과 유지보수성을 확보하도록 설계되었다. 또한 규칙 기반 파싱과 예외적 LLM 복구 구조를 함께 반영하여, 다양한 입력 환경에서도 안정적인 결과를 제공할 수 있도록 하였다.

따라서 본 설계는 MedBuddy를 단순한 이미지 분석 시스템이 아니라, 실제 사용자 환경에서 활용 가능한 복약 보조 시스템으로 발전시키기 위한 적절한 아키텍처 구조라고 판단된다.
